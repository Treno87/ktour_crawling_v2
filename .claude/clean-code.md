# Clean Code Principles

## 명명 규칙

- **의도를 드러내는 이름**: `d`, `elapsedTime` → `elapsedTimeInDays`
- **발음 가능한 이름**: `genymdhms` → `generationTimestamp`
- **검색 가능한 이름**: 매직 넘버 대신 상수 사용
- **클래스명**: 명사/명사구 (예: `Customer`, `WikiPage`)
- **메서드명**: 동사/동사구 (예: `deletePage`, `save`)

## 함수 규칙

1. **작게 만들어라**: 한 함수는 한 가지 일만
2. **한 가지만 하라**: SRP (Single Responsibility Principle)
3. **서술적인 이름 사용**
4. **인수는 적을수록 좋다**: 0개가 이상적, 3개 이상은 피할 것
5. **부수 효과 없이**: 함수 이름에서 예측 가능한 동작만

## 주석

- **코드로 의도를 표현하라**: 주석 대신 명확한 코드
- **좋은 주석**: 법적 주석, 정보 제공, 의도 설명, 경고, TODO
- **나쁜 주석**: 중복, 오해의 여지, 의무적 주석, 위치 표시

## 형식

- **수직 거리**: 관련 있는 코드는 가까이
- **수평 공백**: 연산자 주변에 공백
- **들여쓰기**: 일관되게 유지

## 오류 처리

- **예외를 사용하라**: 오류 코드 대신
- **Null을 반환하지 마라**: Optional, 빈 배열 등 사용
- **Null을 전달하지 마라**

## 클래스

- **단일 책임 원칙 (SRP)**
- **응집도가 높아야 한다**
- **변경하기 쉬워야 한다**

## SOLID 원칙

1. **S**ingle Responsibility: 단일 책임
2. **O**pen/Closed: 확장에는 열려있고 수정에는 닫혀있게
3. **L**iskov Substitution: 하위 타입은 상위 타입을 대체 가능
4. **I**nterface Segregation: 인터페이스 분리
5. **D**ependency Inversion: 의존성 역전

## 리팩토링 타이밍

- 기능 추가 전
- 버그 수정 전
- 코드 리뷰 중
- 중복 코드 발견 시
- 이해하기 어려운 코드 발견 시
